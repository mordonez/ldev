# ldev Architecture Review — Report 2026-05

**Branch:** `feat/architecture-audit-2026-05`
**Date:** 2026-05-13
**Author:** Architecture review agent

---

## Executive Summary

`ldev` started as a Liferay developer CLI tool and has grown into a multi-surface product: a CLI, an MCP server, a Dashboard (HTTP + Preact), and an agentic installer. The May 2026 audit found that the core layered architecture (`cli → commands → features → core`) was sound and had zero directional violations, but two structural smells were accumulating: runtime surfaces (the Dashboard and MCP server) were co-located inside `features/` alongside genuine feature logic, and the naming of `features/mcp/` directly collided with `features/mcp-server/`. Additionally, the output contract system started in R5 covered inventory and resource shapes but did not yet reach MCP tool outputs or dashboard JSON routes.

This session executed all nine recommended changes from the audit in a single branch. The entrypoints were extracted to `src/entrypoints/`, the naming collision was resolved, lint rules were added to enforce layer direction at CI time, bundle boundaries were validated, CONTRIBUTING was updated with the new principles, and Zod schemas were added for all MCP tool outputs and all dashboard HTTP API JSON shapes. The branch exits with `npm run typecheck` clean and ESLint at 9 pre-existing `max-lines` warnings (0 errors, no new warnings introduced).

The overall result is a codebase where a new contributor can immediately distinguish "things ldev does" (features) from "things ldev is" (entrypoints), where layer violations surface as lint errors rather than requiring careful review discipline, and where every JSON shape consumed by an agent or by the Preact client has a versioned Zod contract.

---

## Problems Found

1. **Entrypoints disguised as features.** `features/dashboard/` (24 files, 2,360 LOC) hosted an HTTP server, a task manager, a route dispatcher, and an action catalog. `features/mcp-server/` (21 files, 988 LOC) hosted the MCP stdio server and 17 tool definitions. Both aggregated 8-9 sibling features each. They were categorically different from a feature like `db` or `osgi`.

2. **Naming collision: `mcp` vs `mcp-server`.** `features/mcp/` was the Liferay MCP endpoint probe (a client). `features/mcp-server/` was ldev's own MCP server (a server). A first-time reader picked the wrong directory approximately 50% of the time.

3. **No machine-readable layer enforcement.** The layered architecture (`cli → commands → features → core`) existed only in `CONTRIBUTING.md`. A contributor could import `commander` from a feature or import from `cli/` inside `core/` without any automated signal.

4. **Output contracts incomplete.** `core/contracts/` covered inventory and resource shapes (R5 deliverable) but left MCP tool outputs and dashboard JSON routes without Zod schemas. Silent breaking changes were possible.

5. **Bundle boundaries unverified.** It had never been confirmed whether `commander` leaked into `dist/mcp-server.js` or whether `@modelcontextprotocol/sdk` server-side sub-paths leaked into the CLI bundle.

6. **Dashboard used `localhost` instead of `127.0.0.1`.** On systems where `localhost` resolves to `::1` (IPv6), the Node.js HTTP server binding to `127.0.0.1` was unreachable from the Preact client which requested `http://localhost:port`.

7. **Architectural principles undocumented.** The `entrypoints/` concept, the rule that `core/` must not import from `features/`, and the policy "schema for MCP/dashboard JSON, optional for CLI-only JSON" existed only in the audit document.

## Architectural Decisions

See [ADR 0001 — Stay single-package; do not adopt a `packages/` monorepo (yet)](../adr/0001-monorepo-vs-single-package.md).

**Key decisions made:**

- **Rejected** a `packages/` monorepo split. The boundaries the maintainer wants (enforced surface separation, right-sized bundles) are achievable through directory layout + lint rules at substantially lower tooling cost. Re-open when a genuine external consumer of `core/` appears, the MCP server becomes a separate product, or the dashboard becomes a hosted service.
- **Accepted** `src/entrypoints/` as the new home for non-CLI runtime surfaces (dashboard, MCP server). This is a rename, not a redesign — dependency direction is unchanged.
- **Accepted** `no-restricted-imports` ESLint rules as the enforcement mechanism for layer direction, rather than package walls.
- **Accepted** `features/liferay-mcp/` as the canonical name for the Liferay MCP probe, disambiguating it from ldev's own MCP server.
- **Deferred** entrypoints lint rule (blocking `entrypoints/` from importing `cli/` or `commands/`) until the entrypoint rules are confirmed stable post-move. A comment marks the TODO in `eslint.config.js`.

## Remaining Technical Debt

- **`entrypoints/` import guard not yet active.** The ESLint rule blocking `src/entrypoints/**` from importing `src/cli/**` or `src/commands/**` is commented out in `eslint.config.js`. Estimated effort: 1 hour to enable and validate.

- **Liferay domain internal audit deferred.** `features/liferay/` is 114 files / 20,494 LOC (55% of all feature logic). Whether the internal coupling within `content/`, `inventory/`, `resource/`, `portal/` sub-domains is healthy was explicitly out of scope. Estimated effort: 1 day.

- **CLI `--json` output contracts not formalised.** The approximately 26 commands using `createFormattedAction` produce JSON output whose shape is implied by TypeScript return types only. This is intentional non-scope per audit section 4.2 — apply schemas only where the consumer is an agent or automation.

- **CLI cold-start time not measured.** The audit noted that every `ldev` invocation pays the parse-time cost of importing the dashboard command file. If this becomes a user-facing concern, lazy-loading at the command-registration level is the fix.

- **No `max-lines` violations resolved.** The 9 ESLint warnings for files exceeding 300 lines are pre-existing debt, not introduced by this branch.

---

## Risks

- **The entrypoint moves changed approximately 70 import paths.** While all were mechanical renames and typecheck passed, any integration test importing from the old `features/dashboard/` or `features/mcp-server/` paths would fail at runtime.

- **Lint rule strictness.** The new `no-restricted-imports` rules for `core/` purity and `commands/` layer protection will surface violations in feature branches merged after the audit base commit. This is intended behaviour but is a new failure mode for contributors.

- **MCP SDK client in CLI bundle is intentional but fragile.** The `@modelcontextprotocol/sdk/client/*` sub-paths appear in `dist/index.js` because `ldev mcp doctor` performs a handshake. If a future SDK version exposes server sub-paths via a re-export, the CLI bundle would grow. A bundle check should be run after any MCP SDK upgrade.

- **`features/liferay-mcp/` rename is public CLI-adjacent.** The CLI namespace `ldev mcp ...` was not changed, but `commands/mcp/mcp.command.ts` now imports from `features/liferay-mcp/`. Documentation or scripts referencing the old path `features/mcp/` should be updated.

## Recommended Next Steps

1. **Enable `entrypoints/` import guard** (1 hour). In `eslint.config.js`, activate the commented-out rule blocking `src/entrypoints/**` from importing `src/cli/**` or `src/commands/**`. This closes the last open lint enforcement gap.

2. **Audit Liferay domain internal structure** (1 day). Check whether `content/`, `inventory/`, `resource/`, and `portal/` sub-domains in `features/liferay/` have clean internal boundaries or hidden ordering constraints.

3. **Measure CLI cold-start time** (30 minutes). Run `time ldev --version` ten times and take the median. If P50 exceeds 400ms, profile the startup import graph. The most actionable fix is lazy-loading the dashboard command registration.

4. **Continue contract coverage for CLI `--json` outputs** (2-3 hours). For commands consumed by agents or scripts (e.g., `ldev worktree list --json`, `ldev env status --json`), add Zod schemas to `core/contracts/`.

5. **Split oversized files** (half day). `features/liferay/resource/migration/structure-content.ts` at 824 lines and `pipeline.ts` at 729 lines are the most extreme maintenance risks.

6. **Add MCP SDK version bump guard** (30 minutes). After any upgrade of `@modelcontextprotocol/sdk`, re-run the bundle sanity check (see `docs/architecture/bundle-analysis-2026-05.md`) to confirm server sub-paths have not leaked into `dist/index.js`.
