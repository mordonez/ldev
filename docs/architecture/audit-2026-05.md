# ldev — Architecture Audit (2026-05)

**Scope:** structural review of `src/`, surface boundaries, dependency direction, contracts, build/packaging, and naming. No code changes were made as part of this audit.

**Branch audited:** `main` at `65b5eb5` (single-package layout, two published bins).

**Reviewer perspective:** evaluating whether the current single-package layout still serves the product as `ldev` grows from a CLI into a multi-surface tool (CLI + MCP server + Dashboard + agentic installer), and what to change incrementally without overengineering.

---

## 1. Executive summary

`ldev` is **not under-architected**. The layered model documented in `CONTRIBUTING.md` (`cli → commands → features → core`) is real, enforced in practice, and clean. The dependency direction has zero violations:

- No file in `src/features/` imports from `src/cli/` or `src/commands/`.
- No file in `src/core/` imports from `src/cli/` or `src/commands/`.

The repository is mid-evolution from "CLI" to "product family with multiple surfaces", and the smells that exist today are **boundary smells**, not foundational ones. The right move is to **tighten the existing layout**, not to refactor it into a monorepo. See [ADR 0001](../adr/0001-monorepo-vs-single-package.md) for the packaging decision.

### Top findings

1. **`features/dashboard/` and `features/mcp-server/` are entrypoints living inside the feature layer.** Both host external runtime surfaces (HTTP server, MCP stdio server) and aggregate ~9 other features each. They are categorically different from a feature like `db` or `osgi`.
2. **Output contracts are partially formalised.** `core/contracts/` exists (R5) and covers `inventory` + `resource`. The other ~13 feature domains still emit ad-hoc JSON shapes implied by their return types.
3. **`features/mcp/` and `features/mcp-server/` collide on naming** but mean opposite things — one probes Liferay's MCP, the other hosts ldev's own MCP. The single `commands/mcp/mcp.command.ts` blends both.
4. **The Liferay domain dominates LOC** (114 files / 20.5k LOC, 55% of `features/`). It is already sub-organised by sub-domain (`content`, `inventory`, `resource`, `portal`, …), which is fine, but it is also the part most likely to need its own internal layering as it grows.
5. **There is no machine-readable architectural rule enforcement.** Conventions live in `CONTRIBUTING.md` only — relying on review discipline.

### Top recommendations (prioritised)

| # | Recommendation | Cost | Payoff |
|---|----------------|------|--------|
| 1 | Move `dashboard/` and `mcp-server/` out of `features/` into a new `src/entrypoints/` (or `src/surfaces/`) directory. | Low (rename + import paths) | High — naming finally matches semantics; future entrypoint additions have a home. |
| 2 | Add lint-level enforcement (`eslint-plugin-import` `no-restricted-imports`, or `dependency-cruiser`) for the layer rules. | Low | Medium — prevents the next contributor from regressing layer direction. |
| 3 | Rename `features/mcp/` → `features/liferay-mcp/` (it is the Liferay MCP client probe, not ldev's MCP). | Low | High — kills the worst naming collision. |
| 4 | Continue R5 (contracts) feature-by-feature, **but only for surfaces consumed by MCP or Dashboard**. CLI-only JSON output does not need a versioned schema yet. | Medium (incremental) | Medium — formalises automation contracts where they actually matter. |
| 5 | Document the entrypoint vs feature distinction in `CONTRIBUTING.md`. | Low | Low — locks in the new mental model. |
| 6 | Defer monorepo split. Revisit at v1.0 or when a second consumer of `core/` appears. | — | — |

## 3. Surface boundaries

### 3.1 The two anomalies inside `features/`

Both `dashboard/` and `mcp-server/` are entrypoints, not features. Evidence:

**`features/dashboard/`** imports from **9 sibling features**:

```
features/dashboard/  imports →  db, deploy, doctor, env, mcp-server,
                                oauth, worktree, (+ core)
```

It owns an HTTP server (`dashboard-server.ts`, `dashboard-http.ts`, `dashboard-router.ts`), route modules, an action dispatcher, task management, and a JSON action catalog (`dashboard-actions.json`). It also references its own Preact client under `client/`.

**`features/mcp-server/`** imports from **8 sibling features**:

```
features/mcp-server/  imports →  agent, deploy/deploy-status,
                                 doctor, env/env-logs-diagnose,
                                 liferay/inventory/*, liferay/liferay-health,
                                 mcp/mcp.ts, osgi/* (+ core)
```

It owns the MCP server bootstrap, tool registry, and 17 tool definitions under `tools/`.

A genuine feature returns a domain result. These two **host an external protocol** and aggregate features. They belong adjacent to `cli/`, not inside `features/`.

### 3.2 Proposed grouping

```
src/
├── index.ts                    # CLI entrypoint binary
├── mcp-server.ts               # MCP entrypoint binary
├── cli/                        # Commander framework (unchanged)
├── entrypoints/                # NEW: external runtime surfaces
│   ├── dashboard/              # ← from features/dashboard/
│   └── mcp-server/             # ← from features/mcp-server/
├── commands/                   # CLI command registrations (unchanged)
├── features/                   # Pure feature logic (no entrypoints)
├── core/                       # Shared abstractions (unchanged)
└── testing/                    # Test helpers (unchanged)
```

This is a **rename + move**, not a redesign. The dependency direction stays identical:

```
binary ──> cli ──> commands ──> features ──> core
binary ──> entrypoints/* ──> features ──> core
```

`entrypoints/*` can import `features/*` and `core/*`, but never `cli/` or `commands/`, and never each other.

### 3.3 What this is NOT solving

This move does **not** by itself:

- Reduce dashboard's transitive dependency surface (it still needs the 9 features it aggregates — that is product-defined).
- Reduce MCP server's bundle size (same).
- Stop dashboard deps from being loaded when CLI runs (that is a build-layer concern; see §6).

What it **does** solve: a contributor opening the repo for the first time can immediately distinguish "things ldev *does*" (features) from "things ldev *is*" (entrypoints).

## 4. Contracts and output renderers

### 4.1 Current state

- **`core/contracts/`** has three files (`shared.schema.ts`, `inventory.schema.ts`, `resource.schema.ts`) — the R5 deliverable. Roughly 15 schemas / types exported.
- **`core/output/`** has three files: `formats.ts` (Zod enum `text|json|ndjson`), `printer.ts` (the abstraction), `run-step.ts` (progress UI).
- **Per-feature formatters**: ~20+ feature files export a `formatX(result)` function returning a string. JSON output is `JSON.stringify(result)` — the contract is implicit in the TypeScript return type.

This means **the JSON contract for any command not yet covered by `core/contracts/` is "whatever the feature happens to return"**. That is brittle for two consumers:

1. **The MCP server** — tools return JSON to agents. Without a schema, breaking changes are silent.
2. **The dashboard** — routes return JSON to the Preact client. Same risk.

The CLI itself is more forgiving — humans read `text`; scripts that pipe `--json` are typically internal.

### 4.2 Recommendation

**Do not** try to schema-fy every feature. Apply contracts where the cost of a silent break is real:

- Anything exposed via MCP tools → must have a Zod schema in `core/contracts/`.
- Anything exposed via dashboard JSON routes → must have a Zod schema in `core/contracts/`.
- CLI-only `--json` output → schema is optional, document only if external scripts are known to depend on it.

This is the natural continuation of R5 and matches the principle in the user's prompt: *"todo output JSON debe tener contrato estable"* — but scoped to **public output**, not every internal helper.

### 4.3 Renderers

The pattern "feature returns object + feature exports `formatText`" is fine for now. The temptation to extract a generic `renderers/` layer is **not** justified by the current code — each feature's text rendering has feature-specific logic (column widths, conditional sections) that does not generalise. Resist.

## 5. Naming collisions and concept clarity

### 5.1 `mcp` vs `mcp-server`

This is the worst naming smell in the repo:

- `src/features/mcp/mcp.ts` — **Liferay's official MCP endpoint** probe (check / probe / openapis). It is an MCP **client**.
- `src/features/mcp-server/` — **ldev's own MCP server** for AI agents.
- `src/commands/mcp/mcp.command.ts` — one CLI namespace that mixes both (subcommands `check/probe/openapis` go to `features/mcp/`, subcommand `doctor` goes to `features/mcp-server/`).

A first-time reader will pick the wrong directory ~50% of the time.

**Recommendation:**

```
features/mcp/        →  features/liferay-mcp/      (it is Liferay's MCP, not ours)
features/mcp-server/ →  entrypoints/mcp-server/    (it is an entrypoint, not a feature)
commands/mcp/        →  unchanged for now (public CLI surface)
```

The CLI subcommand grouping under `ldev mcp …` is a different question (it is product-defined). Split CLI namespaces only if user feedback says the conflation is confusing for users — internal naming should be fixed regardless.

### 5.2 Other naming notes

- `features/mcp-server/mcp-server-doctor.ts` vs `features/mcp/mcp.ts` exported `runMcpDoctor` (etc.) — the `mcp-server-` prefix is good, keep it.
- `features/agent/agent-*` vs `features/ai/ai-*` — two adjacent concepts. Worth a short note in CONTRIBUTING: "`ai/` = installer for AI tooling into a project; `agent/` = runtime context exposed to agents". (Not renaming; documenting.)

## 6. Build and packaging

### 6.1 What ships today

```
package.json
  bin:
    ldev              → dist/index.js
    ldev-mcp-server   → dist/mcp-server.js

tsdown.config.ts
  entry: ['src/index.ts', 'src/mcp-server.ts']
  format: ['esm'], platform: 'node', dts: true

vite.dashboard.config.ts
  root: src/features/dashboard/client
  outDir: dist/dashboard-client
```

One source tree, one `package.json`, one set of `dependencies`. tsdown builds both Node binaries; Vite builds the Preact dashboard client separately into `dist/dashboard-client/`.

### 6.2 Implications for bundle weight

Because tsdown bundles per-entry from `src/`, the **MCP server's emitted bundle only contains what `src/mcp-server.ts` transitively imports** — not everything in `src/`. Tree-shaking is doing its job.

This is the central technical fact for the monorepo decision: **package boundaries are not the lever that controls per-binary bundle weight**. Entry-based bundling already is.

The remaining bundle-weight risk:

- If a future feature is imported by *both* CLI and MCP server, it lands in both bundles.
- Heavy deps brought in by *only one* entrypoint (e.g., `node:http` for the dashboard server) do not leak into the other binary as long as no shared module pulls them in unconditionally.

### 6.3 The dashboard's special case

The dashboard server lives inside the CLI binary today (`ldev dashboard` is a subcommand registered in `commands/dashboard/dashboard.command.ts`). That means:

- Starting `ldev` does not start the HTTP server (good — lazy import via the command action).
- But every `ldev` invocation **does** pay the parse-time cost of importing the dashboard command file.

If/when this becomes measurable, the fix is at the command-registration level (lazy-load the command implementation), not at the package level.

### 6.4 `dependencies` review

| Dep | Used by | Notes |
|-----|---------|-------|
| `@modelcontextprotocol/sdk` | MCP server only | Would not leak into CLI bundle. |
| `commander` | CLI only | Should never appear in MCP server bundle. |
| `execa`, `fs-extra`, `p-retry`, `p-wait-for` | Both (likely) | Generic helpers. |
| `jszip` | Dashboard + resource export | Heavy-ish. |
| `picocolors` | CLI rendering | Should not appear in MCP server. |
| `yaml` | Config loading | Both. |
| `zod` | Contracts + schemas | Both. |
| `preact` | Dashboard client (dev) | Already a `devDependency` — built by Vite, not bundled by tsdown. ✅ |

A useful follow-up (not done in this audit) is a one-time `dist/` size check per binary, and a check that `commander` is not in `dist/mcp-server.js`. If it is, that is a real import-graph bug. (Estimated: 5 minutes with `bundle-stats` or `du -sh dist/*.js`.)

## 7. Testability and contributor flow

### 7.1 What works

- Test taxonomy is well-defined (`unit / integration / smoke / system`) and documented in CONTRIBUTING.
- Features take dependencies as parameters — DI-style — which is what makes them testable.
- `src/testing/` exists for shared helpers.
- `npm run verify` is a real local CI gate.

### 7.2 Friction points

- **Adding a new MCP tool currently requires touching three places** (`features/mcp-server/tools/tool-X.ts`, `features/mcp-server/mcp-server-tools.ts` registry, and the underlying feature). Acceptable, but worth documenting the "how to expose a feature via MCP" path in CONTRIBUTING.
- **Adding a new dashboard route** is undocumented. The pattern exists (`dashboard-*-routes.ts`), but a first-time contributor cannot guess it without grep.
- **No machine-readable layer enforcement.** A reviewer must catch `import from '../../cli/'` inside `features/` by eye. Adding ESLint `no-restricted-imports` (or `dependency-cruiser`) would make this automatic.

## 8. Risks and what NOT to do

### Things that look like improvements but are not

| Tempting change | Why it would hurt |
|------------------|-------------------|
| Move to `packages/core`, `packages/cli`, `packages/mcp`, … | See [ADR 0001](../adr/0001-monorepo-vs-single-package.md). High tooling tax, no real coupling gain. |
| Extract a generic `renderers/` layer | Per-feature formatters are not actually generalisable; the abstraction would carry zero shared logic. |
| Schema-fy every feature's JSON | Most CLI `--json` output has no external consumer. Pay the cost where the consumer exists (MCP, dashboard). |
| Rename CLI subcommands to "fix consistency" | Public CLI surface. Breaking changes need product justification, not aesthetic ones. |
| Add an `application/` or `services/` layer between commands and features | Adds a fourth layer for no current need. Most features already are services. |

### Genuine risks of the recommended changes

- **Moving `dashboard/` and `mcp-server/` out of `features/`** changes ~80 import paths. Mechanically safe but produces a large diff. Best done as a single mechanical rename PR with no logic changes, reviewed by file-tree diff rather than line diff.
- **Renaming `features/mcp/` to `features/liferay-mcp/`** is small but touches the public CLI command file. The CLI namespace (`ldev mcp …`) does not need to change.
- **Adding lint rules** will surface existing violations one would not have caught otherwise. Budget half a day for the cleanup.

## 9. Architectural principles (proposed)

These are not aspirational; they describe what the codebase already mostly does. The value is **writing them down** so reviewers can point to them.

1. **One entrypoint per binary**, each living in `src/<name>.ts` and delegating immediately.
2. **`cli/` is the only place that knows about Commander.** No feature, no entrypoint other than the CLI, may import `commander`.
3. **`commands/` is thin.** A command file registers options, parses args, and calls one feature function. No business logic, no I/O orchestration.
4. **`features/` is pure-ish.** Features take config and dependencies as parameters, return data, and may have side effects (HTTP, FS, Docker) at the leaves. No feature imports `cli/` or `commands/`.
5. **`entrypoints/` (new) hosts non-CLI runtime surfaces** (currently dashboard and MCP server). Entrypoints may import `features/` and `core/`. Entrypoints do not import each other and do not import `cli/` or `commands/`.
6. **`core/` is shared infra.** No domain knowledge. Things in `core/contracts/` are versioned in spirit (additive within a major).
7. **Output is feature-local for text, schema-versioned for JSON consumed by automation.** Anything exposed via MCP or dashboard JSON must have a contract under `core/contracts/`.
8. **Errors carry codes.** Per-feature error registries (`features/*/errors/`) already exist for db and deploy. Extend the pattern, do not unify into a global enum.
9. **Destructive commands support dry-run.** Already true for most; make it the default expectation in CONTRIBUTING.
10. **Naming is concrete, not generic.** `liferay-mcp` not `mcp`. `dashboard-task-routes` not `routes`. Avoid `manager`, `handler`, `helper` as suffixes unless there is genuinely no better word.

## 10. What was NOT validated in this audit

In the interest of honesty about scope:

- **Bundle sizes** were not measured. The reasoning in §6 is structural; an actual `du -sh dist/*.js` plus `npx esbuild --analyze` would be needed to claim "MCP bundle does not contain commander".
- **Test coverage of features/dashboard/ and features/mcp-server/** was not assessed.
- **Performance** of CLI cold start was not measured.
- **The 26 commands using `createFormattedAction`** were not individually reviewed for JSON output stability.
- **The Liferay domain (114 files)** was treated as a single block. Its internal substructure (`content/`, `inventory/`, `resource/`, `portal/`) deserves its own audit at some point.
- **Tests were not run.** This audit changed no code.
