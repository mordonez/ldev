# Architecture Review — OSS Readiness

**Date:** 2026-03-28
**Scope:** Full architectural review for open source publication readiness
**Verdict:** Functional project with good internal architecture, not yet publishable as serious OSS

---

## 1. Global Diagnosis

**Status: functional with good technical foundations, but not publishable as serious open source yet.**

### What works well

- Clean layered separation: CLI → Commands → Features → Core → Platform
- Consistent `run*` / `format*` pattern across ~50 commands (logic vs presentation)
- Dependency injection in tests (fake-docker, fake-process, `fetchImpl`)
- 48 test files with proper unit/integration/smoke classification
- Zod config validation, Commander well used, good dependency choices
- Automation contract with `--json`/`--ndjson` is smart design for scripting

### What blocks OSS publication

1. **npm package includes Docker infra, Java modules, AI tools, and scaffolds** — not a CLI package, it's an entire project repo packaged as npm
2. **No linting, no formatting, no CI** — zero automated quality gates
3. **`liferay.command.ts` is 807 lines; `features/liferay/` is 7,500 lines in 41 files** — Liferay API domain dominates and unbalances the architecture
4. **No `.d.ts` generation** — impossible to consume as a library
5. **No CHANGELOG, no real semver, no contributing guide**
6. **MIT license has no author name** — `Copyright (c) 2026` with no holder

**Assessment:** 70% of internal architecture work is solid. The missing 30% (packaging, quality, community readiness) is what separates a functional internal tool from a serious open source project.

## 2. Main Problems

### P1: npm package is an antipattern (CRITICAL)

```json
"files": ["dist", "docker", "liferay", "modules", "scaffold", "tools/ai", ...]
```

Publishing to npm: Docker Compose files, a Java/Gradle OSGi module, scaffolds, AI tools, and migration docs. The tarball will be enormous and confuses the package identity.

### P2: `liferay.command.ts` — 807 lines, God File (HIGH)

Imports 51 features and registers ~30 subcommands in a single function. Any change to any liferay namespace command touches this file.

### P3: Architectural imbalance (HIGH)

```
src/features/liferay/  → 41 files, 7,500 lines (40% of project)
src/features/env/      → 15 files
everything else        → <20 files each
```

### P4: Zero quality gates (HIGH)

No ESLint, no Prettier, no .editorconfig, no CI/CD, no pre-commit hooks, coverage disabled.

### P5: No type declarations (MEDIUM)

`dts: false` in tsup — impossible to consume as TypeScript library.

### P6: Coupled to specific Liferay project (MEDIUM)

Hardcoded image versions, assumed path structures, concrete OSGi module.

### P7: Documentation only in Spanish (MINOR but relevant)

For a public npm project, primary language should be English.

---

## 3. Proposed Architecture

### Model: Single package with clear separation of concerns

No monorepo needed yet. The npm package should contain ONLY JavaScript/TypeScript code.

Docker configs, Java modules, scaffolds, and AI tools should be:
- Generated by commands (`ldev project init` creates Docker structure)
- Downloaded on demand
- Or live in a separate repository

### Dependency flow (already correct)

```
index.ts (entry)
    ↓
cli/ (command parsing & context)
    ↓
commands/ (registration, thin wrappers)
    ↓
features/ (business logic, pure functions)
    ↓
core/ (config, platform, output, http)
    ↓
node_modules (commander, execa, zod, etc.)
```

## 5. CLI / Plugin Strategy

**Verdict: do NOT implement a formal plugin system now.**

- 12 command namespaces, all maintained by same team
- No evidence of external extension demand
- Commander.js has no native plugin system
- Cost of plugin system (discovery, lifecycle, stable API, docs) is high

**Do instead:**
1. Split `liferay.command.ts` into 5 sub-registries
2. Document command addition pattern in CONTRIBUTING.md
3. Leave the door open for future `LdevPlugin` interface if demand arises

## 6. Quality & Tooling Recommendations

- ESLint with typescript-eslint strict
- Prettier with project conventions (bracketSpacing: false, singleQuote: true)
- .editorconfig
- Enhanced npm scripts: lint, format, format:check, check, prepack
- Conventional Commits (documented, not enforced by tooling)
- Coverage enabled with ~70% threshold

---

## 7. Recommended CI

Single `ci.yml` workflow:
- **check job**: lint → format:check → typecheck → test:unit → test:integration (matrix: Node 20, 22)
- **smoke job**: build → test:smoke → binary smoke test
- **package job**: npm pack → size check (<5MB)

Release workflow (add when versioning is real):
- Tag-triggered npm publish with provenance

## 8. Realistic Roadmap

### Phase 1: Required before publishing

1. Clean `files` in package.json — only `dist/` and docs
2. Add ESLint + Prettier, run fix once
3. Add .editorconfig
4. Split `liferay.command.ts` into sub-registries
5. Enable `dts: true` in tsup (if programmatic API desired)
6. Add CI workflow
7. Complete LICENSE with author name

### Phase 2: Recommended (first weeks post-publication)

3. Add CHANGELOG.md
4. Sub-organize `features/liferay/` into subdirectories
5. Rename `core/liferay/` to `core/http/`
6. Add release workflow
7. Installed binary smoke test
8. Document automation contract separately

### Phase 3: Later (when community exists)

1. Evaluate plugin system demand
2. Consider docs site (VitePress)
3. Explore monorepo if separate packages emerge
4. Dependabot / Renovate
5. Security scanning (CodeQL)

---

## 9. What NOT to do now

1. **Formal plugin system** — no demand, high cost
2. **Monorepo / workspaces** — complexity without benefit
3. **commitlint / husky / lint-staged** — CI enforcement on PRs is more pragmatic
4. **Auto-generated docs site** — good manual README > mediocre auto docs
5. **i18n framework** — English messages are fine
6. **DI container** — current parameter injection works
7. **100% coverage** — 70% well-placed > 100% inflated
8. **Semantic release automation** — `npm version` + manual tag is enough
9. **E2E with real Docker** — smoke tests with mocks are sufficient
