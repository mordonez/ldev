# Project Context: ldev

This file is the shared domain and architecture context for contributors and AI agents working on `ldev`.
If this file conflicts with code, the code wins and this file should be updated.

## Product Shape

`ldev` is an operational CLI for Liferay maintenance work. It helps developers and agents:

- understand the current project and portal state
- diagnose local environment, runtime, OSGi, data, and content failures
- reproduce production-like issues in controlled local environments
- apply the smallest safe fix
- verify the result with structured evidence

The product is the operational loop: understand, diagnose, fix, verify. Agent workflows, JSON output, and bootstrap files are support layers on top of that loop, not a separate product story.

Non-goals:

- a generic Liferay CLI
- a plugin platform
- cross-stack tooling beyond Liferay maintenance
- replacing standard Liferay Workspace conventions
- speculative abstractions for hypothetical future adapters

## Domain Language

Use these terms consistently in code, docs, tests, and architecture discussions.

### Liferay Terms

- **Portal**: A running Liferay instance reachable over HTTP.
- **Site**: A Liferay site or group, addressed by friendly URL such as `/global` or by numeric site id.
- **Page**: A Liferay layout under a site, addressed by URL path.
- **Structure**: A web content data definition. In older Liferay names this maps to a DDM structure.
- **Template**: A web content template, usually FreeMarker, often associated with a structure.
- **ADT**: Application Display Template. Display logic for widgets, not a web content template.
- **Fragment**: A page fragment artifact used by the page editor.
- **OSGi bundle**: A deployed runtime unit inspected through Gogo shell and portal diagnostics.
- **Headless API**: Liferay's REST API surface. Prefer it when it exposes the needed operation.
- **JSONWS**: Liferay's legacy JSON web service surface. Use it as a fallback for operations not covered cleanly by Headless APIs.
- **Document Library**: Liferay file and binary content, handled by database and files workflows.

### ldev Terms

- **Project type**: The detected local runtime model: `ldev-native`, `blade-workspace`, or `unknown`.
- **ldev-native**: A project where `ldev` owns the Docker-based local runtime. Detected from `docker/` and `liferay/` layout.
- **blade-workspace**: A standard Liferay Workspace. `ldev` runs on top of it and must not replace its conventions.
- **Runtime adapter**: A module behind the runtime seam for lifecycle actions such as setup, start, stop, status, and logs.
- **Worktree**: An isolated branch checkout, usually under `.worktrees/<name>`, optionally with its own local runtime state.
- **Preflight**: A cached check that validates the API surfaces and credentials needed before longer portal or resource workflows.
- **Inventory**: Structured discovery of portal state, such as sites, pages, structures, and templates.
- **Page evidence**: Normalized searchable references found while inspecting a Page, such as Fragment, Widget, Portlet, Structure, Template, Journal article, or Display Page article evidence.
- **Portal resource**: A Liferay content artifact managed as a stable CLI workflow: structures, templates, ADTs, and fragments.
- **Resource workflow**: A file-based `ldev resource` operation for reading, exporting, importing, or syncing portal resources.
- **Resource migration**: A structured workflow for changing journal structures while preserving or cleaning up existing content.
- **Migration pipeline**: The end-to-end resource migration flow driven by a migration descriptor file.
- **Bootstrap**: Agent-facing project and readiness context generated by `ldev ai bootstrap`.

## Canonical Resource Vocabulary

`ldev resource` uses resource-specific option names instead of generic aliases:

- Structure selectors use `--structure`.
- Template selectors use `--template`.
- ADT selectors use `--adt`.
- `adt --id` is only for the numeric ADT template id; it is not a general alias.
- Template creation uses `--structure` for the related structure key.

Examples:

```bash
ldev resource structure --site /global --structure BASIC
ldev resource template --site /global --template NEWS_TEMPLATE
ldev resource export-structure --site /global --structure BASIC
ldev resource export-template --site /global --template NEWS_TEMPLATE
ldev resource export-adt --site /global --adt SEARCH_RESULTS --widget-type search-result-summary
ldev resource import-structure --site /global --structure BASIC --check-only
ldev resource migration-init --site /global --structure BASIC
```

## Operating Model

Every command should fit one phase of the operational loop:

| Phase      | Purpose                                     | Representative commands                                                                                                          |
| ---------- | ------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| Understand | Resolve project, runtime, and portal state. | `ldev context`, `ldev status`, `ldev portal inventory ...`                                                                       |
| Diagnose   | Localize a failure.                         | `ldev doctor`, `ldev logs diagnose`, `ldev osgi diag <bundle>`                                                                   |
| Fix        | Apply the smallest safe local change.       | `ldev deploy module`, `ldev resource import-* --check-only`, then a deliberate mutation                                          |
| Verify     | Prove the result with fresh evidence.       | `ldev portal check`, `ldev portal inventory ... --json`, `ldev resource structure/template/adt`, `ldev logs diagnose --since 5m` |

Resource verification must be read-after-write. Do not treat log output as sufficient proof that a portal resource changed correctly.

## Architecture

The codebase is layered:

```text
src/
  cli/       CLI framework: context, helpers, output rendering, error normalization
  commands/  Commander registration and option mapping only
  features/  domain workflows and business logic
  core/      shared config, HTTP, output, platform, runtime, contracts, and utilities
  testing/   test helpers only
```

Dependency direction:

```text
commands -> features -> core
commands -> core
```

Rules:

- `core/` must not import from `features/` or `commands/`.
- `commands/` should stay thin: parse options, call one feature workflow, render the result.
- Feature modules should accept dependencies as parameters where tests need fakes.
- Shared behavior belongs in `core/` only when it is genuinely cross-domain.
- A seam is only worth keeping when there are real adapters or meaningful behavior hidden behind it.

## Important Modules

- `src/cli/command-helpers.ts`: formatted command actions and output rendering.
- `src/cli/command-context.ts`: command-time config, printer, cwd, and project context.
- `src/core/config/project-context.ts`: detected repo, project type, profiles, env, inventory, and config.
- `src/core/contracts/`: Zod schemas for shared, inventory, and resource surfaces.
- `src/core/runtime/`: runtime adapter seam for lifecycle behavior.
- `src/features/liferay/`: portal, inventory, resource, page layout, auth, and Liferay-specific operations.
- `src/commands/resource/`: resource command registration split by workflow group.
- `src/features/ai/` and `templates/ai/`: agent-facing install, status, bootstrap assets, skills, and workspace rules.

## Output And Error Contracts

Automation should consume structured output:

- `--json` emits one JSON value.
- `--ndjson` emits one JSON value per line for streaming-capable workflows.
- `--strict` wraps success as `{ "ok": true, "data": ... }`.

All errors should flow through CLI error normalization. Domain error factories provide stable error codes; messages must be safe to display and must not leak secrets.

## Configuration Model

- `.liferay-cli.yml`: versioned, non-secret defaults.
- `.liferay-cli.local.yml`: local non-versioned overrides where applicable.
- `docker/.env`: Docker runtime environment for local Compose projects.
- Environment variables: preferred source for credentials.
- `--repo-root` and `REPO_ROOT`: run against another checkout root without changing directory.
- `ldev portal` and `ldev resource` support Liferay connection overrides for URL, OAuth client id, client secret env var, scope aliases, and timeout.

Do not put plaintext OAuth secrets or passwords in versioned project files.

## Safety Invariants

- Always discover the target project and portal before mutating state.
- Always use `--json` for automation.
- Always run `--check-only` before resource mutations when the command supports it.
- Do not use plural resource mutation commands without explicit human approval.
- Prefer focused commands over broad deploys or broad imports.
- Diagnose failed commands before retrying.
- Verify mutations with operation-specific evidence.

## Testing Lanes

- **Unit**: `tests/unit`, fast pure logic or dependency-injected fakes.
- **Integration**: `tests/integration`, CLI behavior against temp repos without Docker.
- **Smoke**: `tests/smoke`, end-to-end behavior against a real local runtime.
- **System**: `tests/system`, high-I/O package and installed-CLI verification.

Use the narrowest lane that can prove the behavior. Real Docker lifecycle belongs in smoke tests. Installed package checks belong in system tests.

## Release And Packaging

- Package name: `@mordonezdev/ldev`.
- CLI binary: `ldev`.
- Runtime: Node `>=22`.
- Module format: ESM, with `.js` import specifiers in TypeScript source.
- Build: `tsdown`.
- Release management: `release-please` and Conventional Commits.
- Published files include `dist/`, selected `templates/`, docs/readme/license metadata, and packaged AI assets.
